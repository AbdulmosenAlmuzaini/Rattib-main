# وثيقة الأمن السيبراني وحماية البيانات - منصة «رتّب – مكتب المعقّب الذكي»

المنصة ستتعامل مع بيانات شخصية ومستندات حساسة للغاية (مثل بطاقات الهوية، الإقامات، السجلات التجارية، الحوالات والمدفوعات). لذا، فإن تطبيق أقصى معايير الأمان وحماية الخصوصية يقع في صلب تصميم البنية التحتية والبرمجية للمنصة.

---

## 1. نموذج التهديدات (Threat Model)

تم تحديد وتحليل المخاطر الأمنية التالية وتوفير الحلول الوقائية لها:

| التهديد الأمني | الوصف والأثر | استراتيجية الحماية والوقاية |
| :--- | :--- | :--- |
| **تسريب بيانات المستأجرين (Cross-Tenant Leak)** | محاولة مكتب الوصول لبيانات أو مستندات مكتب آخر. | عزل كامل لجميع الاستعلامات البرمجية بالتحقق الإجباري من `workspace_id` على مستوى الخادم (Server-side Row Level Enforcement) وليس عبر الواجهة فقط. |
| **الوصول غير المصرح به (Unauthorized Access)** | دخول مستخدمين غير مخولين أو تعديل الموظفين لبيانات خارج صلاحياتهم. | جلسات خادمية خاصة مخزنة ببصمة SHA-256 داخل PostgreSQL، وكلمات مرور مشتقة عبر scrypt، وملفات ارتباط HttpOnly وSameSite=Strict، مع تطبيق RBAC على الخادم. |
| **رفع ملفات خبيثة (Malicious File Upload)** | قيام مهاجم برفع ملفات تنفيذية خبيثة (.exe, .sh) لاختراق خادم التخزين. | تحديد الامتدادات المسموحة بدقة (PDF, JPEG, PNG, DOCX, XLSX)، فحص الحجم الأقصى، تغيير أسماء الملفات المرفوعة عشوائيًا لتفادي تنفيذها، واستخدام روابط تنزيل مؤقتة وآمنة. |
| **هجمات حقن قواعد البيانات (SQL Injection)** | إدخال نصوص برمجية خبيثة في حقول الإدخال لتنفيذ أوامر بقاعدة البيانات. | استخدام **Prisma ORM** والاستعلامات ذات المعاملات، مع التحقق الخادمي من جميع المدخلات بواسطة Zod. |
| **هجمات البرمجة العابرة للمواقع (XSS)** | حقن شيفرات JavaScript في حقول النصوص (مثل حقل الملاحظات) لتنفيذها في متصفح مستخدم آخر. | تنظيف تام لكافة المدخلات والمخرجات (HTML Sanitization) واستخدام آليات الأمان المدمجة في React للتصيير الآمن للنصوص. |
| **تسريب مفاتيح الأسرار (Secrets Leakage)** | وجود كلمات المرور أو مفاتيح الـ API في الكود المرفوع إلى Git. | استخدام المتغيرات البيئية (`.env`) وتوثيقها بملف `.env.example`. استخدام نظام إدارة الأسرار في الخادم. |

## 2. تصنيف وحماية البيانات الحساسة (Sensitive Data Protection)

يتم تصنيف البيانات في المنصة إلى ثلاثة مستويات:
1.  **بيانات عامة (Public Data)**: اسم مكتب الخدمات، المدينة، اللوجو، التصنيفات العامة. (تخضع لعزل المستأجر العادي).
2.  **بيانات سرية (Confidential Data)**: أسماء العملاء، أرقام الجوال، أسماء الخدمات، سجلات الدفع والمهام.
3.  **بيانات حساسة للغاية (Highly Sensitive Data)**: أرقام الهوية الوطنية، الإقامات، أرقام السجلات التجارية، صور المستندات المرفوعة.

### تدابير حماية البيانات الحساسة للغاية:
*   **الإخفاء الجزئي (Data Masking)**: تُعرض أرقام الهويات والإقامات في جميع القوائم والتقارير العامة مخفية جزئيًا (مثل: `******1234`).
*   **سجل الكشف الأمني (Reveal Audit Trail)**: عندما يقوم موظف مخول بالضغط على زر "إظهار الرقم الكامل"، يتم تسجيل الحدث فورًا في جدول `audit_logs` متضمنًا (اسم الموظف، تاريخ وتوقيت الكشف، معرف العميل، وعنوان IP).
*   **منع تدوين البيانات في السجلات (No-Logging Policy)**: يُحظر تمامًا تدوين أرقام الهوية أو كلمات المرور أو التوكنات أو الهويات داخل سجلات الأخطاء أو Console Logs في الخادم أو العميل.

## 3. الصلاحيات وجدار الأمان (RBAC Policy)

يتم التحقق من الصلاحيات عبر مستويين:
1.  **المستوى الأول (Frontend Guard)**: إخفاء الأزرار والروابط التي لا تتطابق مع دور المستخدم الحالي لتقديم تجربة مستخدم نظيفة.
2.  **المستوى الثاني (Backend Guard - إجباري)**: يقوم الخادم بإعادة التحقق من الجلسة ومن دور الموظف وصلاحياته في قاعدة البيانات قبل تنفيذ أي عملية (إضافة، تعديل، حذف، جلب). لا يتم الوثوق بأي مدخلات قادمة من المتصفح دون فحص خادمي.

---

## 4. حماية المستندات والملفات المرفوعة (Document Security)

*   **عزل الملفات (Storage Isolation)**: يتم تخزين مستندات كل مكتب في مسار معزول خاص به (مثال: `workspaces/workspace_123/documents/`).
*   **التنزيل المصادق عليه**: تُمنع إتاحة روابط الملفات للعامة. يمر تنزيل كل مستند عبر الخادم بعد التحقق من الجلسة والصلاحية والمكتب، ويسجل الحدث في سجل التدقيق.
*   **فحص الامتدادات والأحجام**:
    *   الحجم الأقصى المسموح للملف هو **5 ميجابايت**.
    *   الأنواع المقبولة فقط: `PDF, PNG, JPEG`.
    *   يتم التحقق من نوع الملف الفعلي (MIME Type) وليس فقط من الامتداد الظاهري لاسم الملف.

---

## 5. النسخ الاحتياطي واستعادة البيانات (Backup & Disaster Recovery)

*   **النسخ الاحتياطي التلقائي (Automated Backups)**: تفعيل ميزة النسخ الاحتياطي التلقائي اليومي لقاعدة بيانات PostgreSQL والملفات المرفوعة مع الاحتفاظ بنسخ تاريخية لمدة 30 يومًا.
*   **التشغيل المتعدد والتوافر العالي**: توزيع الخوادم وقاعدة البيانات لضمان عدم توقف الخدمة.
*   **سياسة استرجاع البيانات**: في حال حدوث عطل كارثي، يتم استعادة أقرب نسخة احتياطية آمنة في بيئة معزولة، والتحقق من سلامة البيانات وعزل المستأجرين قبل إعادة توجيه حركة المرور إليها.

---

## 6. التعامل مع الحوادث الأمنية والخرق (Incident Response Plan)

في حال الاشتباه بوجود محاولة اختراق أو خرق أمني:
1.  **العزل المباشر**: إيقاف مساحة العمل المتضررة أو حساب المستخدم المشتبه به فورًا.
2.  **التحقيق البرمجي**: مراجعة سجلات الأمان `audit_logs` للتعرف على مصدر الهجوم والبيانات التي تم الوصول إليها.
3.  **ترقيع الثغرة**: إصدار تحديث أمني فوري لإغلاق الثغرة المكتشفة وتطهير النظام.
4.  **شفافية التواصل**: تنبيه مالك المكتب المتضرر رسميًا بالإجراءات التي تم اتخاذها لحماية بياناته.
